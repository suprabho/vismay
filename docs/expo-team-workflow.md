# Optimized Expo Cloud + Dev Server Workflow for a Small Team

**Audience:** 2‚Äì5 developers, iOS + Android, native packages change regularly.
**Goal:** Minimize rebuild pain, keep JS iteration fast, share builds easily, and get PR previews.

---

## TL;DR ‚Äî The Recommended Setup

1. **Adopt CNG (Continuous Native Generation).** Gitignore `ios/` and `android/`. Treat them as generated artifacts. No more merge conflicts in native files.
2. **Use `expo-dev-client` development builds** instead of Expo Go. One per platform per developer.
3. **Set `runtimeVersion: { "policy": "fingerprint" }`** in `app.json`. Native changes auto-bump the runtime so incompatible OTA updates are impossible.
4. **Enable the EAS remote build cache provider.** When one teammate bumps a native dep and pushes, the next teammate to run `npx expo run:ios` downloads the matching binary instead of rebuilding for 10‚Äì20 minutes.
5. **Use EAS Workflows** to automate dev builds on demand and to publish per-branch EAS Update previews on every push.
6. **Share builds via internal distribution** (URL/QR code) ‚Äî no TestFlight, no Play Console review cycle.
7. **Pin native deps with `npx expo install`** and run `npx expo-doctor` in CI to catch version drift before it breaks a build.

If you do nothing else, do steps 3 + 4. Together they prevent the two biggest team pain points: "my build crashes after the OTA update" and "I have to wait 20 minutes after every `git pull`."

## 1. Project Structure ‚Äî Continuous Native Generation

When you create an Expo project with `npx create-expo-app`, the `ios/` and `android/` directories are auto-generated by `npx expo prebuild` from your `app.json`/`app.config.js` and `package.json`. Treat them like `node_modules`: generated, gitignored, never edited by hand.

**`.gitignore`** (should already contain these on a fresh project):
```
/ios
/android
```

**Why:**
- No native-file merge conflicts.
- Team members don't need Xcode/Android Studio for most work.
- Anyone who needs native changes adds a **config plugin** instead of editing the generated files (which would be overwritten on the next `prebuild --clean`).

**`.easignore`** ‚Äî same syntax as `.gitignore`, but for EAS Build uploads. Use it to exclude large local-only assets that don't need to be sent to the cloud.

## 2. Dev Client + the Local Dev Server

The dev client is your team's own version of Expo Go: a native binary that loads JavaScript from `npx expo start`. Each developer installs it **once per fingerprint change** and reloads JS freely afterwards.

**Install:**
```bash
npx expo install expo-dev-client
```

**Build for each developer (one-time, or after native changes):**
```bash
# Android (works on any OS)
eas build --platform android --profile development

# iOS Simulator (macOS only)
eas build --platform ios --profile development-simulator

# iOS device (requires Apple Developer account)
eas build --platform ios --profile development
```

**Daily workflow after the dev client is installed:**
```bash
npx expo start            # JS bundler ‚Äî fast Metro reload
```

The dev client auto-detects the bundler and connects via QR or Orbit.

**When does the dev client need rebuilding?**
- Adding/removing/upgrading a native package.
- Changing a config plugin or anything under `plugins`/`ios`/`android` in `app.json`.
- Bumping the Expo SDK.
- Changing permissions, app icon, splash screen, bundle ID, or scheme.

Pure JS/asset changes do **not** need a rebuild.

## 3. Runtime Versions ‚Äî Pick `fingerprint`

`runtimeVersion` is the compatibility contract between a build's native layer and the JS update it loads. If they mismatch, the app silently uses an incompatible update and crashes.

**Use the `fingerprint` policy** ‚Äî it hashes your native dependencies, config plugins, and native project files automatically. Add a new lib, the runtime version changes; ship an update against the old runtime, it just won't be delivered to incompatible builds.

**`app.json`:**
```json
{
  "expo": {
    "runtimeVersion": { "policy": "fingerprint" },
    "updates": { "url": "https://u.expo.dev/<your-project-id>" }
  }
}
```

The alternative ‚Äî `"appVersion"` policy ‚Äî works but requires you to remember to bump app version every time you touch native code. With a team, somebody will forget. `fingerprint` removes that footgun.

**Optional but recommended: `fingerprint.config.js`** to skip noise sources:
```js
/** @type {import('@expo/fingerprint').Config} */
module.exports = {
  sourceSkips: [
    'ExpoConfigVersions',                          // marketing version doesn't affect runtime
    'ExpoConfigRuntimeVersionIfString',
    'PackageJsonAndroidAndIosScriptsIfNotContainRun',
  ],
};
```

You can also `.fingerprintignore` files that change for non-native reasons.

## 4. The Single Biggest Speedup ‚Äî Remote Build Cache

When one teammate bumps `react-native-reanimated` and pushes, everyone else needs a new dev client. Without caching, each developer compiles for 10‚Äì20 minutes locally (or waits in the EAS queue).

The **build cache provider** changes that. `npx expo run:ios`/`run:android` first computes the project fingerprint, asks a remote provider whether a binary with that fingerprint already exists, and if so downloads it. If not, it compiles locally and uploads the result for the next person.

**Setup (EAS as provider):**
```bash
npx expo install eas-build-cache-provider --dev
```

**`app.json`:**
```json
{
  "expo": {
    "buildCacheProvider": "eas"
  }
}
```

After this, the typical team flow becomes:
1. Dev A adds a native dep, runs `npx expo run:ios`, compiles for 15 min, binary uploads to EAS.
2. Dev A commits and pushes.
3. Dev B pulls, runs `npx expo run:ios` ‚Üí fingerprint matches ‚Üí downloads in 30 sec.

Custom providers (GitHub Releases, S3, etc.) are also supported if you'd rather not store binaries on EAS.

## 5. `eas.json` Profiles for a Small Team

```json
{
  "cli": { "version": ">= 16.0.0", "appVersionSource": "remote" },
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal",
      "channel": "development"
    },
    "development-simulator": {
      "developmentClient": true,
      "distribution": "internal",
      "ios": { "simulator": true },
      "channel": "development"
    },
    "preview": {
      "distribution": "internal",
      "channel": "preview"
    },
    "production": {
      "autoIncrement": true,
      "channel": "production"
    }
  },
  "submit": { "production": {} }
}
```

`distribution: "internal"` produces an installable APK for Android and an ad-hoc IPA for iOS, with a shareable URL/QR code ‚Äî no app store review needed. Anyone with the link can install.

## 6. EAS Update ‚Äî Fast JS Iteration

EAS Update ships JS-only changes over the air. Two concepts:

- **Branch** ‚Äî a Git-like stream of updates (`development`, `feature/login`, `release-1.2`).
- **Channel** ‚Äî what a build *listens to*. The `channel` field in each `eas.json` build profile sets it.

A branch can be mapped to one or more channels in the EAS dashboard. The clean default for a small team:

| Channel       | Used by                        | Pointed at branch        |
|---------------|--------------------------------|--------------------------|
| `development` | Dev clients                    | current Git branch name  |
| `preview`     | Internal preview builds        | `staging`                |
| `production`  | App Store / Play Store builds  | `production`             |

**Publish a JS update:**
```bash
eas update --branch <branch> --message "fix: empty state copy"
```

**Per-PR preview update (publish to a branch named after the Git ref):**
```bash
eas update --branch ${BRANCH_NAME} --message "$COMMIT_MSG"
```

Every dev client can scan a QR code from the EAS dashboard to load any branch's update ‚Äî useful for design review without anyone pulling code.

## 7. EAS Workflows ‚Äî Cloud Automation

EAS Workflows are YAML-defined jobs that run on Expo's cloud, integrated with your repo. They replace the older "build triggers" feature.

### Workflow A: Create all dev builds on demand

`.eas/workflows/create-development-builds.yml`:
```yaml
name: Create development builds

jobs:
  android_development_build:
    name: Build Android
    type: build
    params:
      platform: android
      profile: development
  ios_device_development_build:
    name: Build iOS device
    type: build
    params:
      platform: ios
      profile: development
  ios_simulator_development_build:
    name: Build iOS simulator
    type: build
    params:
      platform: ios
      profile: development-simulator
```

Run with:
```bash
eas workflow:run .eas/workflows/create-development-builds.yml
```

Or ‚Äî even better ‚Äî let any teammate trigger it from the dashboard or via `eas build:dev`. The three builds run in parallel.

### Workflow B: Publish preview update on every push

`.eas/workflows/publish-preview-update.yml`:
```yaml
name: Publish preview update

on:
  push:
    branches: ['*']

jobs:
  publish_preview_update:
    name: Publish preview update
    type: update
    params:
      branch: ${{ github.ref_name || 'test' }}
```

Now every commit on every branch publishes an update under that branch's name. Open a PR, scan the QR code on the EAS dashboard, the dev client loads the PR's code. No `git pull` required.

### Workflow C (optional): Build on demand from PR labels

Without writing any workflow, the Expo GitHub App lets you add a label like `eas-build-ios:development` or `eas-build-android:preview` to a PR. EAS picks it up and builds. Useful for one-off builds when a PR introduces native changes.

## 8. Handling Native Package Changes ‚Äî The Drill

When you (or a teammate) adds, removes, or upgrades a native dep:

1. **Install with `npx expo install <package>`**, not `npm install`. This pins versions known to be compatible with your Expo SDK.
2. **Commit `package.json` and `package-lock.json`.** No prebuild required ‚Äî `ios/` and `android/` are gitignored.
3. **Don't manually bump `runtimeVersion`.** The fingerprint policy handles it.
4. **Run `npx expo-doctor`** to catch mismatched/unknown native deps before pushing. Add it to your pre-commit or CI.
5. **First person to run `npx expo run:ios` after the change uploads to the build cache.** Everyone else downloads in seconds.
6. **JS-only updates published since the change won't deliver to old dev clients** ‚Äî fingerprint mismatch protects you.
7. **If the change is shipping to staging/production,** trigger a new dev or preview build via EAS (manually or via Workflow A / a GitHub PR label).

**Pre-push checklist for native changes:**
- `npx expo-doctor` clean
- `git status` shows only `package.json`, lockfile, maybe `app.json`/plugins
- Pushed and confirmed via EAS Workflow that a new fingerprinted build is in the cache

## 9. Sharing Builds Across the Team

Three good options, in order of friction:

1. **EAS Dashboard / Email link.** Every internal build gets a permalink and QR code on `expo.dev`. iOS device builds require teammates' UDIDs to be registered (`eas device:create`).
2. **Expo Orbit.** Desktop app that lists every EAS build for your project. One click installs to simulator/emulator/device. The fastest experience for daily iteration.
3. **TestFlight / Play Internal Testing.** Use the `preview` and `production` profiles for stakeholders who don't have Orbit, or for QA on real devices without UDID registration.

For a 2‚Äì5 person team, **Orbit + internal distribution links** is the sweet spot.

## 10. CI/CD with EAS Workflows + GitHub

A pragmatic small-team setup:

| Trigger                              | Action                                                                 |
|--------------------------------------|------------------------------------------------------------------------|
| Push to any branch                   | Publish EAS Update on a branch named after the Git ref (Workflow B)    |
| PR label `eas-build-ios:development` | EAS Build for iOS dev client (catches native-change PRs)               |
| Manual: `eas workflow:run`           | Rebuild all 3 dev clients when fingerprint changes                     |
| Push to `main`                       | EAS Build of `preview` profile + publish update to `preview` channel   |
| Tag `v*`                             | EAS Build of `production` + auto-submit to App Store / Play Store      |

**Why EAS Workflows over GitHub Actions:**
- Native build runners are Expo-managed (no macOS runner billing).
- Tighter integration with EAS Build/Submit/Update ‚Äî no juggling API tokens.
- Free tier covers small teams.

If you already have a heavy GitHub Actions setup, you can also call EAS commands from there via `eas-cli` and an `EXPO_TOKEN`.

## 11. Pitfalls and Mitigations

| Pitfall                                                                  | Mitigation                                                                                 |
|--------------------------------------------------------------------------|--------------------------------------------------------------------------------------------|
| Dev manually edits `ios/Podfile` ‚Äî change vanishes after next prebuild   | Write a **config plugin** instead. Document this in CONTRIBUTING.                          |
| Two devs add the same native dep on different branches                   | Fingerprint policy detects both, but lockfile conflicts must still be resolved manually    |
| Dev forgets to rebuild after pulling a native change                     | Fingerprint runtime mismatch makes the JS bundle refuse to load ‚Üí loud, obvious error      |
| Anonymous config plugin functions don't change fingerprint when edited   | Use **named** functions in separate files (see fingerprint docs)                            |
| Big `node_modules` slow down EAS Build uploads                           | `.easignore` to exclude unneeded files; EAS auto-caches `node_modules` by lockfile hash    |
| Per-PR preview updates pollute dashboard                                 | Configure branch retention in EAS dashboard; archive old branches periodically             |
| iOS device build needs UDID                                              | Run `eas device:create` once per teammate, then their UDID is in the team's provisioning   |

## 12. The 30-Second Mental Model

Picture two layers:

- **Native binary** ‚Äî what's compiled into the IPA/APK. Changing this requires a rebuild. The dev client is one of these.
- **JS bundle + assets** ‚Äî what `npx expo start` serves locally and what EAS Update ships OTA. Changing this is instant.

`runtimeVersion` is the contract: a JS bundle only runs on a binary with a matching runtime. Set it to `fingerprint` and the contract is enforced automatically.

Three commands cover 90% of the work:
```bash
npx expo start                 # daily JS iteration
eas build --profile development # rebuild dev client when fingerprint changes
eas update --branch <name>     # ship JS to teammates / preview / production
```

Add the build cache provider so step 2 is mostly a download, not a compile, and you've eliminated the two slow loops in Expo team development.
